Feature Engineering

Now that we've acquired a comprehensive understanding of the dataset, we can proceed with feature engineering.

Feature engineering is a pivotal stage in the machine learning process, as the quality and relevance of the features can exert a profound influence on the model's performance and its ability to provide accurate predictions or classifications. This endeavor combines the art of creativity, domain expertise, and the science of leveraging a deep comprehension of both the data and the specific problem at hand.

After many iterations, 12 features were chosen to train our ML model:

Original Features:

  1. 'expected_goals'
  2. 'influence'
  3. 'goals_scored'
  4. 'assists'
  5. 'bps'
  6. 'xP'
  7. 'bonus'
  8. 'minutes'
  9. 'creativity'
  10. 'threat'
  11. 'clean_sheets'

Feature Interactions:

  1. Product of 'bps' and 'goals_scored'
In[3]:
def feature_engineering(data):
    
    # Define Features
    data['bps_goals'] = data['bps'] * data['goals_scored']
    
    feature_selected = ['expected_goals', 
                        'influence',
                        'goals_scored',
                        'assists',
                        'bps', 
                        'xP', 
                        'bonus',
                        'minutes',
                        'creativity',
                        'threat',
                        'bps_goals',
                        'clean_sheets'] 

    selected_data = data[feature_selected]
    
    return selected_data

Training on Historical Data and Testing on the Current Season

Let's try training on Seasons 2020-2021 to 2023-2024, and predicting on Season 2024-2025. This is a common practice in FPL and sports models, because analysts want to determine whether historical trends apply to and can be used to predict current performance. Also, we were very curious ourselves and wanted to confirm how our model performs for the current season!

In[15]:
train = master_cleaned[(master_cleaned['Season'] != '2024-2025')]
test = master_cleaned[(master_cleaned['Season'] == '2024-2025')]

X2 = feature_engineering(train)
y2 = train['total_points']

X3 = feature_engineering(test)
y3 = test['total_points']

# Instantiate a classifier model
model2 = RandomForestRegressor(n_estimators=100, max_depth = 15, min_samples_leaf=2, min_samples_split=10, random_state=42)

# Perform cross-validation (e.g., 5-fold cross-validation) on the training set
cv_results2 = cross_val_score(model2, X2, y2, cv=5, scoring='r2')  # Use appropriate scoring metric
print("Cross-validation R^2 scores:", cv_results2)
print("Mean R^2 score:", cv_results2.mean())

# Train on X2 and y2
model2.fit(X2, y2)

# Test the model on the test set
y_pred3 = model2.predict(X3)
test_r2 = r2_score(y3, y_pred3)

# Evaluate the test performance
print("Test R^2 score:", test_r2)
print("Test Mean Squared Error (MSE):", mean_squared_error(y3, y_pred3))
print("Test Mean Absolute Error (MAE):", mean_absolute_error(y3, y_pred3))

In general, we can see that our model performs very similarly/consistently across the 2 data splits. The second split (which tests on the current season) does exhibit a slightly lower test 'R2' score (0.944) and slightly higher MSE, albeit with a slightly reduced MAE. This indicates that on average, the errors have become smaller and the model is making better predictions, but that the predictions for some outliers may have worsened, which is why we were further penalized on MSE.

This can be attributed to the difference in the amount of training and testing data. While the original split was 80:20, the second split tested on only season 2024-2025, which was only 2 months into an 8 month run (much less testing/more training data than in the original split).

However, the change in error statistics and 'R2' scores is relatively minor, indicating solid model performance in both cases.
